# Code Review Guidelines

这个文档定义了我们项目的Code Review标准和要求。

## Code Review 重点关注领域

### 1. 代码质量
- **命名规范**: 变量、函数、类名应该清晰描述其用途
- **代码复杂度**: 函数应该保持简洁，单一职责
- **注释**: 复杂逻辑必须有清晰的注释说明
- **代码重复**: 避免重复代码，提取公共方法

### 2. 安全性检查
- **输入验证**: 所有外部输入必须进行验证和sanitization
- **认证授权**: 确保API端点有适当的权限检查
- **敏感信息**: 不允许硬编码密码、token等敏感信息
- **SQL注入**: 使用参数化查询，避免字符串拼接SQL

### 3. 性能优化
- **数据库查询**: 避免N+1查询问题，合理使用索引
- **内存使用**: 及时释放不再使用的资源
- **异步处理**: 耗时操作应该异步执行
- **缓存策略**: 合理使用缓存减少重复计算

### 4. 错误处理
- **异常捕获**: 所有可能出错的地方必须有适当的错误处理
- **错误日志**: 关键错误必须记录详细日志便于排查
- **用户友好**: 错误信息应该对用户友好，不暴露系统内部信息
- **降级处理**: 关键功能应该有降级方案

## Review 严重程度分级

### Critical (严重) - 必须修复
- 安全漏洞
- 数据丢失风险
- 系统崩溃风险
- 性能严重问题

### Warning (警告) - 强烈建议修复
- 代码质量问题
- 潜在的bug
- 性能优化机会
- 测试覆盖不足

### Info (信息) - 建议考虑
- 代码风格问题
- 更好的实现方式
- 文档改进建议
- 重构机会

## 不需要Review的情况

- 纯文档更新（如README、注释）
- 配置文件的简单修改
- 测试数据的更新
- 日志格式的调整
